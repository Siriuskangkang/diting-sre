# 数据库连接池耗尽排查

## 现象
应用日志大量出现 "Cannot get a connection" / "Connection pool exhausted" / "Wait timeout"，接口大面积超时或报错，但数据库本身 CPU/负载并不高。

## 常见根因
1. 连接泄漏：手动获取的连接（非框架托管）未在 finally 中 close，连接被耗尽不归还。
2. 长事务 / 慢 SQL 长期占用连接，池中可用连接被占满。
3. 连接池上限设置过低，无法支撑当前并发。
4. 批量任务一次申请大量连接，挤占在线业务。
5. 连接因数据库重启或网络抖动变为"脏连接"，校验失败后被反复重建。

## 排查步骤
1. 看连接池监控：活跃连接数(active)是否等于最大上限(maxActive)，等待线程数(wait)是否堆积。
2. dump 应用线程栈，看大量线程是否卡在 `getConnection`。
3. 在数据库侧查当前会话：`SHOW PROCESSLIST`，看是否有大量来自该应用的慢会话。
4. 审查代码中手动获取连接的位置，确认都有 try-with-resources / finally close。
5. 评估当前并发与池上限是否匹配（经验值：单实例 20~50，按 QPS × 平均耗时估算）。

## 修复方案
- 泄漏：补全连接归还逻辑，启用连接泄漏检测（如 Hikari 的 leakDetectionThreshold）。
- 慢 SQL：定位并优化，缩短连接占用时间。
- 上限不足：调大 maxActive，但注意数据库侧 max_connections 总量。
- 临时：重启应用释放连接（治标），再定位根因。

## 预防措施
- 统一用框架托管事务，禁止裸 JDBC 手动管理连接。
- 启用连接泄漏检测和池满告警。
- 连接池参数配合压测调整，预留余量。
