# HTTP 5xx 错误率升高排查

## 现象
监控面板显示某服务 5xx 错误率在短时间内从 0.1% 飙升到 5% 以上，外部调用方开始报"服务不可用"或"上游错误"。

## 常见根因
1. 下游依赖（数据库 / 缓存 / 第三方接口）不可用或超时，导致网关返回 502/504。
2. 服务自身抛未捕获异常（空指针、反序列化失败），返回 500。
3. 容器 OOM 被 kill 后重启，请求期间返回 5xx。
4. 配置/密钥刚发布有误，启动期大量请求失败。
5. 限流或熔断器误开启，把正常请求当作故障拒绝。

## 排查步骤
1. 先看错误类型分布：是 500 居多（自身 bug）还是 502/504 居多（下游/网关）。
2. 对齐时间线：是否与最近一次发布、配置变更、扩缩容重合（查 CI/CD 部署事件）。
3. 查该服务日志，grep 异常堆栈关键词，确认是否单一异常类型集中爆发。
4. 查下游依赖健康度：数据库连接数、Redis 延迟、第三方接口成功率。
5. 查容器指标：是否伴随 OOMKilled 或频繁重启（`kubectl get pods` 看 RESTARTS）。

## 修复方案
- 下游问题：切备用实例 / 降级走本地缓存 / 触发熔断保护上游。
- 自身 bug：快速回滚到上一个稳定版本，再定位修复。
- 资源问题：临时扩容（加副本 / 加内存），再排查内存泄漏。

## 预防措施
- 关键接口加熔断 + 降级 + 超时三件套。
- 发布走灰度，配合自动告警和一键回滚。
- 对下游依赖做健康检查和自动故障转移。
