# 接口延迟突增（P99 飙高）排查

## 现象
某核心接口 P99 从 80ms 涨到 800ms 以上，整体成功率未明显下降，但用户体验明显卡顿，可能触发上游超时级联。

## 常见根因
1. 慢查询：数据库出现全表扫描或缺失索引的 SQL，单次查询耗时数百 ms。
2. 锁竞争：热点行被长事务持有，其他请求排队等锁。
3. 资源瓶颈：CPU 打满、GC 频繁（Full GC 停顿）、网络带宽跑满。
4. 批量任务挤占：大查询 / 报表任务 / 数据同步在业务高峰跑，抢占资源。
5. 连接池耗尽：请求排队等可用连接，表现为整体延迟均匀上升。

## 排查步骤
1. 区分"少数请求极慢"还是"所有请求都变慢"：前者多是个别慢查询/热点，后者多是资源/池子问题。
2. 查 APM 链路：定位耗时集中在 DB / 缓存 / 下游 RPC 哪一段。
3. 查慢查询日志和数据库当前执行计划，确认是否新增慢 SQL 或索引失效。
4. 查 GC 日志和 CPU/内存水位，确认是否有 Full GC 或 CPU 被打满。
5. 查连接池监控：活跃连接数是否接近上限、是否有等待。

## 修复方案
- 慢查询：临时 kill 长查询，补索引或加 LIMIT，长期优化 SQL。
- 锁竞争：定位并中断长事务，拆分热点写。
- 资源瓶颈：扩容、错峰跑批任务、限制并发。
- 连接池：调大上限或排查连接泄漏。

## 预防措施
- 核心接口配置 P99 告警和容量基线。
- 慢查询阈值告警 + DBA 评审机制。
- 大批量任务错峰执行，与在线业务资源隔离。
