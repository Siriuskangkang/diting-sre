# 产品定位

## 名字由来

**谛听（DiTing）**：中国神话中地藏菩萨的神兽，传说能「听辨世间万物真假、识一切根由」。

- **契合产品本质**：听辨故障、识根因——正是排障 AI 要做的事
- **文化记忆点强**：在满是 "X-Copilot / X-Bot" 的市场里有辨识度
- **命名友好**：英文 `DiTing` 简短好发音，可直接做代码命名（`diting-api` / `diting-agent`）

> 备选曾考虑：夜鹰 NightHawk（on-call 夜值守）、哨兵 Sentinel（守护）、灯塔 Lighthouse（指引）。

## 定位

**SRE / DevOps 的 AI 排障自动驾驶仪。**

谛听**不是**"运维版 ChatGPT"（被动问答），而是一个**会主动介入、会调工具取证、会自我进化**的排障 Agent。分工类比：

| 工具 | 职责 |
|---|---|
| PagerDuty / 夜莺 | 负责报警 |
| Prometheus / Datadog | 负责监控 |
| **谛听** | **负责"接警后自己查清楚怎么回事、怎么修"** |

## 目标用户画像

**主用户：中型科技公司的 SRE / DevOps 工程师（on-call 一线）**
- 痛点：半夜被告警叫醒、排障靠翻 wiki 和问老员工、知识断层（核心人员离职就断档）
- 期望：告警一响，AI 已经查得差不多了，自己只需确认和决策

**决策者：运维 / 平台团队 Tech Lead、CTO**
- 关心：MTTR（故障平均恢复时间）、知识沉淀、降低对个别"老员工"的依赖、可审计

## 四个核心场景（详述）

### 场景 1：On-call 故障自排查（核心 · MVP 聚焦）

- **触发**：告警系统（Prometheus Alertmanager / 夜莺 / PagerDuty）通过 webhook 推送告警到谛听
- **谛听自动**：启动排障 Agent → 调监控确认指标异常 → 查日志抓异常堆栈 → 查 Pod/容器状态 → 搜知识库（历史同类故障 runbook）→ 搜 GitHub 已知 issue → 综合推理
- **产出**：在飞书 / Slack / 钉钉群**实时播报**排查进展，最终给「根因 + 证据链 + 修复方案 + 预防措施」
- **价值**：MTTR 大幅下降；on-call 不再从零开始；新人也能独立排障

### 场景 2：运维知识库（基础能力 · 已实现）

- 多格式文档（PDF 运维手册、Word 操作规范、故障截图 OCR、Markdown runbook）一键入库（百炼解析）
- 带引用溯源的问答（答案标注出处 chunk，可核查、防幻觉）
- **价值**：团队知识从"散落在 wiki / 飞书 / 人脑"变成"可推理、可复用的资产"

### 场景 3：新人 Onboarding

- 新人遇到告警或问题，直接问谛听，得到**基于团队历史 runbook 的标准排查路径**
- 不用反复打扰老员工，缩短新人上手周期

### 场景 4：故障复盘 & Runbook 自动化（进阶 · V2）

- 每次排障自动生成结构化复盘报告（时间线 / 根因 / 修复 / 预防 / 责任人）
- 进阶：读 runbook **自动执行**修复动作（kubectl 回滚 / 扩容 / kill 慢查询），用 LangGraph `interrupt` 做人工审批节点
- **价值**：从"建议"到"行动"，进入运维自动化的深水区

## 价值主张

| 对象 | 核心价值 |
|---|---|
| 一线 SRE | 告警一响，AI 已查清；不再半夜抓瞎翻手册 |
| 团队 Lead | MTTR 下降、知识沉淀、降低单点人员依赖 |
| 组织 | 运维知识资产化，人走知识留 |

## 差异化（三组对比）

### vs 通用 AI 助手（ChatGPT / 通义 / 智谱清言）

| | 通用助手 | 谛听 |
|---|---|---|
| 触发 | 人主动提问 | **告警自动触发** |
| 数据 | 无私有数据 | 团队私有知识库 + 真实基础设施 |
| 可信 | 易编造 | 引用溯源、证据链 |
| 行动 | 只输出文本 | 调工具取证、可执行修复 |

### vs 传统 AIOps（Datadog / Splunk / SkyWalking）

| | 传统 AIOps | 谛听 |
|---|---|---|
| 强项 | 检测 + 告警 + 看板 | **根因推理 + 自主取证 + 修复建议** |
| 根因定位 | 仍靠人对照看板 | Agent 自主交叉验证 |
| 关系 | 互补（谛听接它的告警/数据） | 补上"诊断大脑" |

### vs 只读 RAG 知识库产品

| | 只读 RAG | 谛听 |
|---|---|---|
| 检索 | 写死在链里 | Agentic RAG（检索是工具之一） |
| 知识维护 | 靠人手动 | **排障结论自动沉淀（自进化）** |
| 推理 | 单轮问答 | 多步推理 + 工具调用 |
