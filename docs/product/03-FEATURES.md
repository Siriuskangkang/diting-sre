# 功能规划

## 一、MVP（P0 · 约 4 周）—— 打通「告警 → 自排查 → 播报」闭环

| 模块 | 功能 | 当前状态 |
|---|---|---|
| 告警接入 | 接收 Prometheus Alertmanager / 夜莺 webhook 告警 | 待做 |
| 排障 Agent | 告警触发自动启动 LangGraph 多 Agent 排查 | **原型已通** |
| 工具集成 | 接 Prometheus（指标）+ Loki/ELK（日志）+ k8s（Pod 状态）+ GitHub issue | 部分原型（mock）|
| 知识库 | 多格式入库（PDF/Word/图片 OCR）+ 引用溯源问答 | **已实现** |
| IM 播报 | 排查过程 + 结果推送飞书 / Slack / 钉钉 | 待做 |
| Web 控制台 | 知识库管理 + 排障工作流可视化 | **原型已通**（OD 设计稿）|

**MVP 验收标准**：真实 Prometheus 触发告警 → 谛听 60 秒内在 IM 群给出「根因 + 证据链 + 修复方案」，且答案可溯源到知识库文档。

## 二、V1（P1 · 上线后约 3 个月）—— 自进化 + 多源 + 评估

- **知识自进化闭环**：排障结论自动总结成新 runbook 入库（**核心壁垒，优先级最高**）
- **复盘报告**：自动生成结构化 incident 复盘（时间线 / 根因 / 修复 / 预防 / 责任人）
- **更多数据源**：接 Grafana / Datadog / 云厂商监控 / Jira 工单 / GitLab
- **MCP 工具市场**：标准化工具接入协议，用户与三方可贡献 MCP server
- **效果评估看板**：排障准确率、MTTR 改善幅度、知识库覆盖率

## 三、V2（P2 · 6–12 个月）—— 自动执行 + 企业级

- **Runbook 自动执行**：读 runbook 自动执行修复（kubectl 回滚 / 扩容 / kill 慢查询），用 LangGraph `interrupt` 做人工审批
- **多租户 + 权限**：RBAC、团队隔离、完整审计日志
- **SSO + 合规**：企业 SSO、敏感数据脱敏、私有化部署
- **根因知识图谱**：跨 incident 的「症状 → 根因」图谱，复用历史经验，避免重复踩坑

## 四、功能矩阵（完整视图）

| 能力域 | MVP | V1 | V2 |
|---|---|---|---|
| 告警接入 | Prometheus / 夜莺 | Grafana / Datadog / 云监控 | 全渠道 + 智能去重 |
| 排障 Agent | 单故障自排查 | 多故障并发 + 协作 | 跨团队协同 |
| 工具调用 | 监控 / 日志 / k8s | 工单 / GitHub / 云 API | **执行类（带审批）** |
| 知识库 | 多格式入库 + 问答 | **自进化沉淀** | 知识图谱 |
| 交付 | IM 播报 | + 复盘报告 | + 自动执行 |
| 平台 | 单机 Web 控制台 | 多用户 | 多租户 / SSO / 审计 |

## 五、核心用户故事

1. **作为 on-call SRE**，我希望告警一响，谛听就在 IM 群开始直播排查进展——这样我不用半夜爬起来从零查。
2. **作为团队 Lead**，我希望每次排障的根因和修复自动沉淀成 runbook——这样知识不随人员流失。
3. **作为新人**，我希望遇到陌生告警时问谛听就能得到标准排查路径——这样不用反复打扰老员工。
4. **作为 CTO**，我希望看到 MTTR 下降的数据和所有排障的审计记录——这样可度量、可合规。

## 六、优先级与依赖关系

```
P0（MVP 必须）：
  告警接入 → 排障 Agent → 工具集成 → IM 播报
                  ↑
            （知识库已就绪，供 Agent 检索）

P1（差异化壁垒，最高优先）：
  知识自进化闭环（依赖排障 Agent 稳定）→ 复盘报告 → MCP 工具市场

P2（商业放大）：
  Runbook 自动执行（依赖审批 + 权限）→ 多租户 / SSO / 审计
```

**取舍原则**：MVP 只做「单条告警端到端跑通 + IM 播报」这一条主线，砍掉一切旁支（多租户/执行/图谱都进 V1/V2）。先把「告警一响，AI 已查清」这个 demo 效果做到极致——它是最强的获客与面试展示点。
